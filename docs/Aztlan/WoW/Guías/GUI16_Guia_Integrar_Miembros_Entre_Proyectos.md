# Guía para reasignar e integrar miembros entre proyectos

## Propósito
Establecer una forma para identificar cuándo un team member de un proyecto puede integrarse a otro proyecto departamental, cómo decidir la mejor asignación posible y cómo realizar su incorporación minimizando bloqueos y tiempo de adaptación.

Esta guía se recomienda cuando existen varios proyectos activos simultáneamente, algunos equipos avanzan o terminan antes que otros y/o un proyecto requiere apoyo para resolver una necesidad específica sin comprometer a otros proyectos. Se espera que la existencia de esta guía evite tener que mover personal de manera improvisada.

---

## Desarrollo

### Paso 1: Identificar oportunidad de reasignación
Detectar si existe una situación en la que un team member pueda apoyar en otro proyecto sin comprometer el avance de su proyecto de origen. Para identificar una oportunidad de reasignación, se puede apoyar en los siguientes criterios:

- El proyecto ya concluyó su MVP, MBI1 y MBI2
- El team member terminó sus actividades de desarrollo en su proyecto, según el PVG
- El proyecto presenta sobrecarga de trabajo, retrasos, bloqueos o falta de alguna habilidad específica a poco tiempo de la fecha de entrega acordada
- Se detecta una desviación considerable entre el plan y el progreso real que requiera una acción inmediata ([PRC14 - Proceso para Seguimiento de Proyectos](#))

### Paso 2: Confirmar estado del proyecto de origen
Antes de proponer la integración de un team member a otro proyecto, se debe validar que su salida parcial o total no afecte a su proyecto de origen. Se recomienda confirmar que:

- El team member no tiene tareas de alta prioridad pendientes
- El equipo no depende de las skills del team member para cerrar tareas con alta prioridad
- El cambio no pone en riesgo acuerdos definidos con el equipo, departamento o socio formador
- Los PMs y el team leader correspondiente están de acuerdo con liberar capacidad

Este paso se relaciona con el [PRC10 - Proceso para el ajuste de trabajo y recursos](#), el cual profundiza en la necesidad de reajustar el plan y los recursos sin comprometer la viabilidad del trabajo comprometido. 

### Paso 3: Analizar la necesidad del proyecto destino
Una vez detectada la disponibilidad de un team member, se debe definir qué necesidad específica existe en el proyecto destino.  Se debe identificar:

- Qué problema se busca resolver con la incorporación
- Si la necesidad es temporal o permanente
- Qué tarea o entregable se encuentra afectada
- Qué habilidad, conocimiento o tipo de apoyo hace falta
- El impacto que tendría no incorporar apoyo adicional

### Paso 4: Comparar habilidades disponibles y requeridas
Con base en la necesidad del proyecto destino, se debe analizar si el team member disponible cuenta con las habilidades requeridas para aportar valor. Se debe también determinar si el team member puede integrarse de manera inmediata, si requiere acompañamiento, o si necesita una breve capacitación antes de participar (revisar [PRC20 - Proceso para Identificar Habilidades Requeridas](#)). Más adelante se profundiza en el acompañamiento y el contexto mínimo para los team members a integrar.

### Paso 5: Definir el tipo de integración
Para evitar ambigüedad sobre disponibilidad y responsabilidades, se recomienda elegir un tipo de integración específico desde el inicio, de acuerdo con la necesidad del proyecto destino y la disponibilidad del team member:

- Apoyo para una tarea o entregable en específico
- Apoyo temporal durante un sprint específico
- Asignación parcial alternando trabajo entre el proyecto de origen y el proyecto destino
- Reasignación total hasta el cierre del proyecto destino

### Paso 6: Definir responsables para la incorporación
Al realizar una integración de team members, se debe llevar un registro de responsables. Se recomienda definir:

- Quién propone la reasignación
- Quién autoriza la liberación del miembro
- Quién recibe al miembro en el proyecto destino
- Quién lo acompaña durante su adaptación
- Quién debe ser informado del cambio

Este registro puede documentarse dentro de la [Plantilla de Registro de Integración entre Proyectos](../Plantillas/PLA23_Registro_Integracion.md) en la sección 4 (responsables de la integración).

### Paso 7: Transferir contexto necesario
Para que el team member pueda incorporarse sin problemas, se requiere que reciba contexto indispensable del proyecto destino, aunque ya exista conocimiento previo gracias a las reuniones intradepartamentales y a los reportes de estado. 

Idealmente, el team member ya conoce:
- Objetivo del proyecto destino
- Estado actual del proyecto destino
- Documentación del proyecto destino

Y se recomienda compartir:
- Tareas con alta prioridad
- Riesgos, dependencias o bloqueos
- Repositorios y herramientas necesarias
- Acuerdos con el socio formador

La información transferida debe estar documentada y ser accesible, favoreciendo la colaboración y reduciendo dependencias innecesarias. Para más información, revisar  [Política de colaboración](../Políticas/Politica_de_Colaboración.md).

### Paso 8: Asignar una primera tarea controlada
*(Opcional)*

Para facilitar la adaptación, se recomienda asignar una primera tarea al nuevo team member con las siguientes características:

- Debe aportar valor al proyecto
- Debe permitir al miembro familiarizarse con el flujo y las herramientas
- Puede servir como evaluación inicial de la integración
- No debe ser de prioridad alta o aislada del resto del equipo

### Paso 9: Seguimiento a la integración
Después de la integración, se debe revisar si la reasignación está siendo productiva con las siguientes verificaciones:

- El team member ya comprendió el contexto necesario
- El team member logró avanzar en las tareas asignadas
- No surgieron bloqueos técnicos o de comunicación
- La necesidad del proyecto destino sí se está atendiendo
- El tipo de integración elegido fue correcto o necesita ajustarse

Este seguimiento puede realizarse dentro de las sesiones de trazabilidad y seguimiento de proyectos. Más información en [PRC14 - Proceso para Seguimiento de Proyectos](#).

Si en el seguimiento se identifican bloqueos, estos deben ser tratados conforme a la [Política de Gestión de Bloqueos y Dependencias](#).

---

## Registro de integración
Toda reasignación debe dejar un registro que permita dar trazabilidad a la decisión tomada y facilitar el aprendizaje en futuras integraciones entre proyectos. Este registro deberá realizarse cada vez que se apruebe una reasignación, independientemente de si el apoyo será puntual, temporal, parcial o total.

Para ello, se insta a utilizar la siguiente plantilla:

[Plantilla de Registro de Integración entre Proyectos](../Plantillas/PLA23_Registro_Integracion.md)
