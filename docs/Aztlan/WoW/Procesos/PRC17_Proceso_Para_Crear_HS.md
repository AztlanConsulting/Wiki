## Propósito
Tener un proceso claro para la creación de Historias de Usuario que permita a los equipos crear historias de usuario de calidad, con criterios de aceptación claros y que cumplan con las necesidades del cliente.

## Entradas
- Wireframes
- Criterios de aceptación
- Requisitos definidos

## Descripción
Es importante tener en cuenta que una buena historia de usuario sigue la estructura de las "3 C's": Card (Tarjeta), Conversation (Conversación) y Confirmation (Confirmación/Criterios de Aceptación). 


| Fases | Actividades | Prácticas asociadas al CMMI |
|--------------|--------------|--------------|
| Identificar y Definir el Usuario (Persona) | Antes de escribir, es necesario saber para quién se construye la funcionalidad. <br/> Crear Personas: Define personajes arquetípicos (ej. "Administrador", "Cliente Final") que representen a los usuarios reales. Identificar necesidades: Entender qué valor buscan obtener del producto. |    | 
| Redactar la Historia de Usuario **(La Tarjeta - Card)** | Utiliza un formato estándar para asegurar que la historia sea clara y breve: <br/> <br/> **Como** [rol del usuario], <br/> **quiero** [realizar una acción/funcionalidad], <br/> **para** [obtener un beneficio/valor]. <br/> <br/> _Ejemplo: Como cliente registrado, quiero guardar productos en mi lista de deseos para poder comprarlos más tarde._ <br/> <br/> Consejo: Mantén las historias pequeñas y atómicas (una sola funcionalidad por historia). |    | 
| Establecer Criterios de Aceptación **(La Confirmación - Confirmation)** | Son las condiciones específicas que la historia debe cumplir para ser considerada "terminada" (Done). Funcionan como pruebas de aceptación. <br/> <br/> Utiliza un formato como "Dado que" (contexto), "cuando" (acción) y "entonces" (resultado). Ayudan al equipo a entender qué constituye el éxito. |    |  
| Conversar sobre la Historia **(La Conversación - Conversation)** | La tarjeta es solo un recordatorio; la verdadera definición ocurre mediante el diálogo entre el Product Owner y el equipo de desarrollo. Se discuten detalles, reglas de negocio y restricciones técnicas. Se aclaran dudas y se refinen los criterios de aceptación. |    |
| Validar con **INVEST** | Para asegurar la calidad de la historia, aplica el modelo INVEST: <br/><br/> Independent (Independiente) <br/> Negotiable (Negociable) <br/> Valuable (Valiosa) <br/> Estimable (Estimable) <br/> Small (Pequeña) <br/> Testable (Demostrable/Testeable) |    |
| Priorizar el Backlog (SVG) | Una vez definida, la historia de usuario se añade al Product Backlog (SVG) y se prioriza basándose en valor, riesgo y complejidad. <br/><br/> Valor: Qué tanto beneficia al negocio o al usuario. <br/> Riesgo/Complejidad: Historias con alto valor y bajo riesgo suelen priorizarse primero. |    |

\
Resumen de los componentes clave:\
\
Rol: Quién\
Objetivo: Qué\
Beneficio: Por qué\
Criterios de Aceptación: Cuándo está terminada\

## Salidas
- Lista de Historias de Usuario listas para estimación (Base para el RM SP 1.2).
