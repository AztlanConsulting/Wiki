## Propósito

Establecer, definir y gestionar los cambios que se generen dentro de los requisitos durante el desarrollo del proyecto. Así mismo asegura que haya evaluaciones, documentación y control adecuado, incluyendo la identificación y resolución de inconsistencias entre requisitos y productos de trabajo.

## Entradas

- Requisitos
- Solicitudes de cambio
- Historias de usuario
- Diagramas
- Diseños de UI/UX
- Matriz de trazabilidad (RTM)

## Descripción

| Fase | Actividades  | Prácticas asociadas al CMMI |
|-----|-------------|-----------------------------------------|
| Identificación | Identificar inconsistencias entre requisitos y productos de trabajo (historias de usuario, diagramas, diseños, documentación). Registrar la inconsistencia en el **Registro de inconsistencias**. |  REQM SP 1.5 |
| Solicitud de cambio | Registrar formalmente la solicitud de cambio cuando se detecta una inconsistencia o nueva necesidad del proyecto. | REQM SP 1.3 |
| Evaluación | Revisar la solicitud de cambio para validar que esté claramente definida, justificada y alineada al proyecto. | REQM SP 1.3 |
| Análisis de impacto | Analizar el impacto del cambio en alcance, costo, cronograma, arquitectura, riesgos y otros requisitos relacionados. | REQM SP 1.3 |
| Aprobación | Presentar el análisis al responsable o comité de cambios para decidir si el cambio se aprueba, rechaza o se pospone.  | REQM SP 1.3 |
| Documentación | Actualizar la documentación oficial: especificación de requisitos, matriz de trazabilidad, control de versiones y registro de cambios. | REQM SP 1.3 |
| Comunicación | Informar al equipo de desarrollo y a los interesados sobre el cambio aprobado para asegurar su correcta implementación. | REQM SP 1.4 |
| Verificación de consistencia | Verificar que los productos de trabajo actualizados sean consistentes con los requisitos modificados. Registrar cualquier nueva inconsistencia detectada.  | REQM SP 1.5 |
